# 🎵 Treeble: Минималистичный музыкальный плеер для Linux

## 🎯 1. Концепция и Философия
* **Идея:** «Файловый менеджер для музыки». Дерево папок слева, список треков справа.
* **Очередь:** Папка = очередь. Кнопки Next/Prev переключают треки внутри текущей папки.
* **Отказ от лишнего:** Нет потокового аудио, нет БД медиатеки, нет плейлистов, нет скробблинга, нет обложек, нет эквалайзера. Только локальные файлы.

## 🏗️ 2. Общий Стек
* **Backend:** C++ (с прицелом на будущий GStreamer).
* **WebView:** Saucer (использует WebKitGTK на Linux).
* **Frontend:** React + TypeScript + Vite + shadcn/ui.
* **IPC (Inter-Process Communication):** Симметричный JSON с полем `type`.

## ⚙️ 3. Backend (C++)

### Архитектура аудио-слоя
* **Абстракция:** Интерфейс `IAudioBackend` (методы `load`, `play`, `pause`, `seek`, `position`, `duration`).
* **Текущая реализация:** `WebViewAudioBackend` (управляет HTML5 `<audio>` внутри WebView).
* **Будущая реализация:** `GStreamerAudioBackend` (нативный пайплайн, переключается одной строкой).

### Логика и Состояние (Source of Truth)
* **Вся бизнес-логика в C++:** Backend хранит текущую очередь (`std::vector<Track>`), индекс текущего трека и статус. Frontend — «тупой терминал», только отрисовывает UI и шлёт сырые команды.
* **Загрузка файловой системы:** Гибридный подход. Дерево папок строится при старте (быстро). Содержимое папок (треки) грузится лениво при клике и кэшируется в `std::unordered_map`.

### Метаданные
* **Библиотека:** TagLib.
* **Данные:** Только Title, Artist, Duration.
* **Fallback:** Если теги пустые или битые — используется имя файла. Обложки (Cover Art) игнорируются.

### Интеграция с ОС (MPRIS2)
* **Библиотека:** GDBus (использует системный GLib/GIO, который уже есть из-за WebKitGTK).
* **Объем:** Стандартный плеер (Play/Pause/Next/Prev, Position, Seek, Volume).
* **Исключения:** Интерфейсы `TrackList` и `Playlists` не реализуются (сохраняем минимализм).

### Мост к WebView (IPC)
* **Backend -> Frontend:** Прямые вызовы JS-функций через `saucer::app::eval("window.audioPlayer.play()")` с обязательным экранированием путей.
* **Frontend -> Backend:** Единая точка входа через `saucer::app::bind("audioEvent", ...)`. Все события (`ended`, `timeupdate`, `error`) приходят в виде JSON-объектов с полем `type`.

### Стриминг локальных файлов
* **Проблема:** Песочница WebKitGTK блокирует прямые пути `file:///`.
* **Решение:** Кастомная URI-схема `app://audio/...`. C++ бэкенд перехватывает запросы, читает файл чанками и стримит байты в WebView. Поддерживает HTTP-заголовки `Range` для корректной работы перемотки (seek) без загрузки всего файла в RAM.

## ⚛️ 4. Frontend (React + Vite)

### State Management
* **Библиотека:** Zustand.
* **Причина:** Идеально подходит для обновления состояния из глобальных IPC-обработчиков (когда C++ шлёт событие `timeupdate`, Zustand обновляется напрямую, без танцев с Context API). Отлично интегрируется с экосистемой shadcn/ui.

### UI Компоненты
* **Стиль:** shadcn/ui (минималистичный, нативный вид, темная/светлая тема).
* **Структура:** 
  * Сверзу: базовые кнопки управления, полоска прогресса, ползунок громкости.
  * Слева: `TreeView` (папки).
  * Справа: `DataTable` или `List` (треки текущей папки с подсветкой активного).
  * Снизу: статус-бар с информацией о количестве файлов в текущем каталоге и имя проигрываемого трека.

## 📦 5. Сборка и Дистрибуция

### Build System
* **Мастер-сборщик:** CMake.
* **Процесс:** CMake находит системные зависимости (TagLib, GIO), вызывает `npm run build` (Vite) для сборки фронтенда, а затем встраивает папку `dist/` прямо в C++ бинарник как ресурсы (через макросы Saucer).

### Распространение
* **Формат:** AppImage.
* **Стратегия:** Бинарник + ассеты упаковываются в один `.AppImage` файл. 
* **Зависимости:** WebKitGTK **не** бандлится внутрь (чтобы не раздувать размер до 500+ МБ), а берется из системы пользователя (`libwebkit2gtk-4.1`). В README указывается одна команда для установки зависимости. Это сохраняет нативность, легковесность и простоту запуска.
